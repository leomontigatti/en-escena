# Organize app code by product surface

We will organize UI and route code first by product surface (`admin`, `portal`, `auth`, `audit`, `judging`) and only then by domain resource. Admin and portal views often share domain entities but differ in permissions, copy, states and allowed actions, so shared code should stay at field primitives or domain services rather than full cross-surface screens.

Event-scoped rules and master data are described in the domain as Bases del evento. Concrete modules should use direct resource names inside their product surface; for example, admin category routes live under `app/features/admin/categories/`. Older `event-*` feature folders are legacy names to migrate as those verticals are touched. Broader `event-bases` names are reserved for aggregators that genuinely coordinate several of those resources.

`app/features` owns product experiences: route loaders/actions, request and form parsing, redirects, pending states, UI copy, route-level views and flow-specific tests. Feature modules should be organized by product surface first (`app/features/admin/categories/`, `app/features/portal/choreographies/`) because the same domain entity often has different permissions and workflows in each surface.

`app/lib` owns reusable behavior behind small interfaces: domain rules, persistence-oriented modules, access policy, storage adapters and cross-surface orchestration. Domain-neutral modules should use direct domain names (`app/lib/categories/`, `app/lib/choreographies/`, `app/lib/events/`). Surface-specific `app/lib/admin/*` and `app/lib/portal/*` modules are allowed only when the behavior is reusable across multiple flows in that surface and is not just a route adapter moved out of `features`.

Do not move code into `app/lib` merely because it is server-side or shared by files inside one feature folder. If a module knows about a specific screen, route action, `ActionData`, redirects, notification query parameters, form pending state or Spanish UI copy, it belongs in the feature surface or in a legacy aggregator until it can be split. If a module can be exercised through a small interface without knowing which screen called it, it belongs in `app/lib`.

Within `app/lib`, prefer filenames that name the capability or interface they expose (`registration-resolution.server.ts`, `dancer-records.server.ts`, `verification.ts`, `category-identity.ts`) over catch-all domain filenames such as `categories.server.ts` once the module starts collecting unrelated behavior. A broad facade is acceptable only when it stays small and stable; implementation should move behind more specific modules.

Migration should be incremental. When touching an existing vertical, first choose the correct ownership, then shrink the interface before moving code. Do not relocate a whole file into `app/lib` if it still imports feature paths, route action types, redirects or notification helpers. Keep legacy aggregators such as `event-bases` in place while they coordinate several resources, and drain resource-specific behavior out of them as each resource is actively changed.

For Bases del evento persistence, prefer resource-level interfaces for new callers: categories should be reached through `app/lib/categories/`, modalities through `app/lib/modalities/`, schedules through `app/lib/schedules/` and prices through `app/lib/prices/`. Existing `app/lib/events/bases-repository*` modules may remain as implementation or compatibility facades during migration, but new direct imports should be limited to aggregators and migration work. Callers that need several resources coordinated should use `app/lib/events/bases.server.ts`.
